How should the calculator engine run in the browser?

GitHub Pages can only serve static files — no Python runtime, no server. Three ways to bridge the gap:

Option A: Pyodide (Python → WASM)

Compile the full Python scientific stack (sympy, astropy, numpy) to WebAssembly. The existing AstroCalculator class runs natively in the browser with minimal code changes.

  • Keep 90%+ of existing Python code
  • Full astropy constants/units database
  • Exact same results as CLI version
  • ~25-40MB initial download (cached after first visit)
  • 3-8 second cold-start load time
  • Heavier than native JS

Option B: Port to TypeScript

Rewrite the expression parser in TS, bundle a precompiled constants database as JSON, and implement unit conversion manually for the supported unit set.

  • Fast load (< 1MB, instant)
  • No Python dependency
  • Better TypeScript tooling (tree-shaking, static types)
  • Lose astropy's full unit system
  • Must reimplement implicit multiplication parser
  • Ongoing maintenance burden to keep constants/units in sync

Option C: math.js + Custom Extensions

Use the math.js library (which already has an expression parser, implicit multiplication, and unit support) and extend it with astro-specific constants and units.

  • Expression parser + units built in
  • Well-maintained, good docs
  • Moderate bundle size (~200KB gzipped)
  • Unit system is less sophisticated than astropy
  • Mismatch in results vs. Python CLI version
  • Still need to map all constants manually

Recommendation

I'd suggest Option A (Pyodide) despite the larger download. The user base (astronomers) is patient with load times if the results are trustworthy. Keeping the exact same astropy constants and units means calculations produce the same numbers as the CLI — this is critical for scientific credibility. The 25-40MB WASM payload is cached by the browser, so it's a one-time cost per update.
