Design — Section 6: Offline & Local Use

Two modes: local development (faster, always offline) and deployed with service worker (offline after first visit).

Option 1: Local Dev Server (fastest)

npm run dev                    # vite dev — loads Pyodide from local disk
# → http://localhost:5173          # fully offline, near-instant Pyodide load

Pyodide WASM files are pulled from node_modules/pyodide/ — no network at all. Load time drops to ~1-2s instead of 3-8s.

Option 2: Production Build, Served Locally

npm run build                  # vite build → dist/
npm run preview                # or: python -m http.server -d dist/
# → http://localhost:8080        # fully offline, optimized build

Option 3: Deployed + Service Worker (offline after first visit)

Add the Vite PWA plugin (vite-plugin-pwa) with a service worker strategy. On first visit at the GitHub Pages URL, the service worker caches:

  • All app JS/CSS/HTML (~200KB)
  • Pyodide WASM (~30MB — cached as a single chunk)
  • Constants/units JSON (~50KB)

Subsequent visits: instant load, fully offline. The service worker serves everything from cache. The app only re-fetches when you ship a new version (cache-busted by filename hashes).

Recommended setup

For daily use: run the local dev server — fastest Pyodide load, zero network.

For sharing / quick access: use the GitHub Pages URL with service worker. First visit is slow (~5s), every visit after is instant and offline-capable.

All three modes work. Local is the best experience. The deployed version gets close after the first load. Does this cover what you need?
